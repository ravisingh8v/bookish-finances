1) Persistent offline data across app restarts

If I open the app tomorrow while still offline, it should still show meaningful data.

Minimum cached data:

latest 10 recent books
inside each book cache at least 20 recent expenses
cached dashboard summary
cached totals and analytics
recent filters / last viewed book
pending unsynced operations queue

This data must persist across:

refresh
browser close
next day reopen
device restart

Use IndexedDB as primary storage, not session/local state.

2) Full offline CRUD support

Offline mode must support:

create
edit
delete
restore / undo recent delete
mark paid / unpaid
category changes
notes updates
book updates

Everything should work without internet errors.

All offline actions should be stored in a sync queue.

3) Reliable sync engine

When internet comes back:

sync queued actions in order
retry failed sync automatically
prevent duplicate expense creation
resolve conflicts safely using:
latest updated timestamp
server merge fallback
never break totals after sync
analytics must remain accurate
4) Dashboard offline support

Dashboard should still work offline with cached:

total expenses
category totals
book-wise totals
recent spending trend
monthly summaries
last synced timestamp

Even if some old data is archived, recent summaries must stay accurate.

5) Smart large-data strategy for browser limits

Because expense data can grow large, use:

lazy loading + pagination
cache only recent data deeply
archive old data as summarized monthly chunks
store full details only for recent books
keep dashboard aggregates precomputed
use background cleanup for stale cache

Recommended approach:

recent active books → full cache
old books → summary cache only
old expenses → monthly compressed summary
sync queue → highest priority persistence

This avoids browser storage issues.
